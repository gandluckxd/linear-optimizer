# Headless runner для оптимизации москитных сеток

Runner принимает `GRORDERS_MOS.ID`, а не `ORDERID`: по этому ID API получает
связанные СЗ из `GRORDER_UF_VALUES` (`USERFIELDID = 8`). Он использует тот же
workflow, что и GUI: загрузка профилей/материалов, линейный и плоскостной
раскрой, запись карт, затем складские документы и номера ячеек.

## Конфигурация и запуск

Скопируйте `client/linear_optimizer_mos.example.txt` рядом с EXE как
`linear_optimizer_mos.txt`, укажите `api_url`, затем выполните:

```powershell
mos-optimizer-runner.exe 123 --dry-run
mos-optimizer-runner.exe 123
```

Безопасная проверка из исходников (не создаёт ни карт раскроя, ни складских
документов):

```powershell
cd client
py mos_optimizer_runner.py 123 --dry-run --config linear_optimizer_mos.txt
```

`--dry-run` загружает данные и запускает оба оптимизатора, но не вызывает ни
одного API-метода записи. В обоих режимах stdout содержит один JSON-объект с
итогом. Журнал и старые диагностические `print` выводятся только в stderr и в
`log_file` из конфигурации. В итоговом JSON есть количество деталей, карт и
отход, предупреждения, а также `documents.outlay_id` и `documents.supply_id`.

TXT открывается и редактируется обычным Блокнотом. Это плоский файл без
разделов: каждая настройка задаётся как `имя = значение`. Строки с `#` или `;`
в начале — комментарии; комментарий также можно поставить после значения.
Логические параметры принимают `да`/`нет`, `true`/`false` или `1`/`0`; дробные
числа можно писать через точку или запятую. Поддерживаются UTF-8, UTF-8 BOM и
Windows-1251 (ANSI). Ошибка указывает номер неправильной строки.

Чтобы использовать TXT-файл из другого места, передайте его явно:

```powershell
mos-optimizer-runner.exe 123 --config D:\Altawin\linear_optimizer_mos.txt
```

Коды завершения: `0` — успех, `2` — аргумент/конфиг, `3` — загрузка через API,
`4` — оптимизация, `5` — запись результатов/склад/ячейки, `10` — неожиданная
ошибка. При любой ошибке stdout также содержит JSON с `success: false`,
`stage`, `grorders_mos_id` и `error`.

## Складские документы и повторный запуск

Перед расчётом runner вызывает `GET /api/mos-job-state/{id}` и выбирает режим:

- `mode=fresh` — активных складских документов нет; выполняется расчёт и
  создаётся одна пара `OUTLAY`/`SUPPLY`;
- `mode=resume` — карты, детали и корректная складская пара уже существуют;
  runner не пересчитывает и не перезаписывает карты, а только дополняет
  `CELL_NUMBER` и проводит недостающие документы;
- `mode=noop` — карты, ячейки и оба проведённых документа уже полностью
  проверены; никаких операций записи нет;
- `mode=inconsistent` — дубли, половина пары, документы без карт или
  несовпадение содержимого. Runner возвращает контролируемую ошибку
  `warehouse/resume` и ничего автоматически не удаляет.

API связывает документы с заданием точными комментариями:

```text
Списание материалов для оптимизации москитных сеток: <GRORDERS_MOS.ID>
Приход деловых остатков от оптимизации москитных сеток: <GRORDERS_MOS.ID>
```

`adjust-materials-altawin` блокирует создание пары на уровне задания и перед
INSERT повторно ищет действующие документы. Корректная существующая пара
возвращается с `reused=true`; повторные строки `OUTLAYREMAINDER` не
добавляются. Проводка идемпотентна: заголовок с `ISAPPROVED=1` пропускается,
черновик проводится штатным обновлением заголовка, которое запускает
существующие Firebird-триггеры. Runner никогда не исправляет `RESERVEQTY`
напрямую.

После timeout или разрыва соединения runner перечитывает состояние. Если
операция уже зафиксирована, выполнение продолжается; неоднозначное состояние
останавливает workflow без повторного INSERT.

## Безопасное развёртывание

Порядок обязателен, потому что новый runner требует новые read/approve
эндпоинты:

1. Остановить автоматический повторный запуск старого runner.
2. Развернуть API и перезапустить его процесс.
3. Проверить `GET /api/mos-job-state/<тестовый-id>` и убедиться, что ответ
   содержит `optimized_count`, `detail_count`, `missing_cell_count`,
   `outlays`, `supplies` и `warehouse_content_matches`.
4. Запустить database-free тесты из раздела ниже.
5. Собрать и развернуть новый `mos-optimizer-runner.exe`.
6. Сначала выполнить один контролируемый запуск и проверить в JSON/логе
   `mode`, `OUTLAYID`, `SUPPLYID` и итоговое состояние.

Для частичного задания 263 ожидается `mode=resume` с `OUTLAYID=19024` и
`SUPPLYID=12679`. Не удаляйте эти документы и не освобождайте резерв вручную:
после развёртывания совместимых API и runner повторный запуск должен заполнить
135 отсутствующих `CELL_NUMBER`, затем провести существующий расход и приход и
завершиться итоговой проверкой.

## Автотесты без Firebird

```bash
python3 -m unittest discover -s client/tests -p "test_*.py" -v
python3 -m unittest discover -s api/tests -p "test_*.py" -v
```

Оба набора используют fake API/соединения и не открывают рабочую базу.

## Сборка Windows EXE

Запускать на Windows, из корня репозитория:

```powershell
py -m pip install -r client\requirements-headless.txt
py -m PyInstaller --noconfirm --clean --distpath dist\windows --workpath build\windows linear-optimizer-headless.spec
Copy-Item client\linear_optimizer_mos.example.txt dist\windows\linear_optimizer_mos.txt
```

Результат: `dist\windows\mos-optimizer-runner.exe`. Рядом с ним должны лежать
`linear_optimizer_mos.txt`; папка `logs` создаётся автоматически. macOS-сборка не
может заменить Windows `.exe`.
